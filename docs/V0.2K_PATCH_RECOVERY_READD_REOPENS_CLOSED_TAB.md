# V0.2K Patch — Recovery Re-add Reopens Closed Tabs

## Purpose

This patch improves Recovery Journal behavior.

Previously, closed-tab recovery exposed two separate actions:

- Reopen URL
- Re-add to Workspace

That worked, but it created an overlap problem. If the recovery source was a closed browser tab, the user reasonably expects Re-add to Workspace to restore the workspace record and reopen the browser tab.

## Behavior

Done when:

- Re-add to Workspace still restores removed workspace records.
- If the recovery source event is `browser_tab_closed_and_removed`, Re-add to Workspace also reopens the saved URL as a browser tab.
- The reopened browser tab metadata is written back into the restored workspace record.
- Existing workspace records are refreshed instead of duplicated when the same `workspaceTabId` is already present.
- Recovery events preserve `workspaceTabId` when available.
- The System Journal records `workspace_tab_readded` with `browserTabReopened` metadata.
- Repeated Re-add attempts produce a terminal `workspace_tab_readded` event instead of causing action-result diagnostics to time out.

## Implementation

The fix is implemented as a side panel recovery override module:

```text
src/sidepanel/recovery-combined-restore.js
```

The module intercepts the Recovery Journal `Re-add to Workspace` button in capture phase, before the older handler runs.

## Design Rule

Recovery actions now behave as follows:

```text
Removed from workspace, browser tab still open:
- Re-add restores the workspace record.

Closed browser tab and removed from workspace:
- Re-add restores the workspace record.
- Re-add also reopens the saved URL as a browser tab.

Already restored:
- Re-add refreshes the existing record and records a terminal recovery event.
```
