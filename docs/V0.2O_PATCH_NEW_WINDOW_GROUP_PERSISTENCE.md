# V0.2O Patch — New Window Group Persistence

## Purpose

This patch refines `Move Workspace Into New Window` so the action preserves Chrome Flow's cognitive grouping model after moving tabs to a new browser window.

Validation showed that tabs moved correctly, but native Chrome tab groups did not persist across the window move. The Operator had to manually click `Create Chrome Tab Groups` again. A follow-up validation also showed that creating a new window directly from an existing workspace tab could be visually fragile.

## Updated Behavior

`Move Workspace Into New Window` now performs this safer sequence:

```text
1. Resolve open workspace tabs with the stable one-to-one resolver.
2. Sort tabs by workspace role order.
3. Create a normal empty Chrome window first.
4. Wait briefly for the new window to settle.
5. Move all resolved workspace tabs into that already-open new window.
6. Remove the temporary empty tab Chrome created for the new window.
7. Refresh workspace tab metadata.
8. Recreate role-based Chrome tab groups in the new window.
9. Refresh workspace tab metadata again so new groupIds are stored.
10. Record one coherent workspace_tabs_moved_to_new_window event.
```

## Stability Guard

The move action now has an in-flight guard:

```text
Move Workspace Into New Window is already running. Please wait for it to finish.
```

This prevents duplicate move requests while Chrome is still settling the window and tab movement operation.

## Event Evidence

The event `workspace_tabs_moved_to_new_window` should now include:

```text
newWindowCreationMode: empty_window_then_move_all_tabs
temporaryTabIds: [...]
removedTemporaryTabIds: [...]
recreatedChromeGroups: true
recreatedGroupCount: <number>
groupedTabCount: <number>
groups: [...]
```

## User-Facing Result

After clicking `Move Workspace Into New Window`, the Operator should see:

- a fully opened new Chrome window
- all resolved workspace tabs moved into that window
- the temporary new-tab page removed after workspace tabs move in
- workspace role groups recreated automatically
- no need to manually click `Create Chrome Tab Groups` again

## Validation Checklist

1. Start from a workspace with several assigned roles.
2. Click `Create Chrome Tab Groups`.
3. Confirm groups exist.
4. Click `Move Workspace Into New Window` once.
5. Confirm a normal Chrome window opens fully.
6. Confirm all open workspace tabs move into that new window.
7. Confirm the temporary empty tab is removed.
8. Confirm Chrome groups are automatically recreated in the new window.
9. Click `Refresh Workspace Tab Metadata`.
10. Confirm all workspace tabs show `grouped`.
11. Copy workspace and diagnostic packets.
12. Confirm `workspace_tabs_moved_to_new_window` includes `newWindowCreationMode: empty_window_then_move_all_tabs`.
13. Confirm `workspace_tabs_moved_to_new_window` includes `recreatedChromeGroups: true`.
14. Confirm final diagnostic status has `groupedTabs` equal to `totalTabs` and `pendingActionTraces: []`.

## Integration Note

This remains part of the separate advanced controls layer. Do not integrate advanced controls natively until the reattached advanced layer validates cleanly above the restored base controller.
