# V0.2O Patch — New Window Group Persistence

## Purpose

This patch refines `Move Workspace Into New Window` so the action preserves Chrome Flow's cognitive grouping model after moving tabs to a new browser window.

Validation showed that tabs moved correctly, but native Chrome tab groups did not persist across the window move. The Operator had to manually click `Create Chrome Tab Groups` again. That disrupted the intended cognitive flow.

## Updated Behavior

`Move Workspace Into New Window` now performs this sequence:

```text
1. Resolve open workspace tabs with the stable one-to-one resolver.
2. Sort tabs by workspace role order.
3. Create a new Chrome window using the first resolved workspace tab.
4. Move the remaining workspace tabs into the new window.
5. Refresh workspace tab metadata.
6. Recreate role-based Chrome tab groups in the new window.
7. Refresh workspace tab metadata again so new groupIds are stored.
8. Record one coherent workspace_tabs_moved_to_new_window event.
```

## Event Evidence

The event `workspace_tabs_moved_to_new_window` should now include:

```text
recreatedChromeGroups: true
recreatedGroupCount: <number>
groupedTabCount: <number>
groups: [...]
```

## User-Facing Result

After clicking `Move Workspace Into New Window`, the Operator should see:

- a new Chrome window
- all resolved workspace tabs inside that window
- workspace role groups recreated automatically
- no need to manually click `Create Chrome Tab Groups` again

## Validation Checklist

1. Start from a workspace with several assigned roles.
2. Click `Create Chrome Tab Groups`.
3. Confirm groups exist.
4. Click `Move Workspace Into New Window`.
5. Confirm a new Chrome window opens.
6. Confirm all open workspace tabs move into that new window.
7. Confirm Chrome groups are automatically recreated in the new window.
8. Click `Refresh Workspace Tab Metadata`.
9. Confirm all workspace tabs show `grouped`.
10. Copy workspace and diagnostic packets.
11. Confirm `workspace_tabs_moved_to_new_window` includes `recreatedChromeGroups: true`.
12. Confirm final diagnostic status has `groupedTabs` equal to `totalTabs` and `pendingActionTraces: []`.

## Integration Note

This remains part of the separate advanced controls layer. Do not integrate advanced controls natively until the reattached advanced layer validates cleanly above the restored base controller.
