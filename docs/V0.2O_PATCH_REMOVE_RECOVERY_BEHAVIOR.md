# V0.2O Patch — Remove and Recovery Behavior

## Purpose

This patch resolves a behavior mismatch in the tab-management layer.

Validation showed that `Remove from Workspace` removed the Chrome Flow workspace record, but left the live browser tab and native Chrome group behind. This caused later Recovery actions to report that the tab was already open or already grouped.

A second overlap appeared when using:

```text
Close Browser Tab
Reopen URL
Re-add to Workspace
```

The recovered tab was correctly reopened and re-added, but if Chrome Flow created a new role group for the recovered tab, that group could be unnamed.

## Runtime Placement

Current side panel load order:

```text
diagnostics.js
sidepanel.js
workspace-session-control.js
search-workspace-intake.js
remove-workspace-tab-cleanup.js
recovery-group-restore.js
advanced-tab-controls.js
advanced-new-window-stability-v2.js
```

## Remove from Workspace Behavior

A new companion module handles the `Remove from Workspace` button:

```text
src/sidepanel/remove-workspace-tab-cleanup.js
```

It intercepts only `.remove-tab-button` and changes the behavior to:

1. Confirm removal.
2. Prompt for a reason.
3. Capture a recovery snapshot.
4. Close the live browser tab when safely resolved.
5. Remove the workspace record.
6. Record `workspace_tab_removed` with `browserTabClosed` evidence.
7. Preserve `canReopenUrl` and `canReaddToWorkspace` recovery actions.

This keeps the diagnostic terminal event type stable as `workspace_tab_removed` while making the browser state match the workspace state.

## Close Browser Tab Behavior

`Close Browser Tab` remains native and unchanged.

Expected behavior:

- closes the browser tab
- removes the workspace record
- Chrome naturally removes the closed tab from its Chrome group
- Recovery Journal can reopen/re-add later

## Reopen URL Behavior

`Reopen URL` remains native and unchanged.

Expected behavior:

- reopens the saved URL only
- does not add the tab back to the workspace
- does not restore grouping

## Re-add to Workspace Behavior

`recovery-group-restore.js` now handles two extra cases:

1. If Re-add follows Reopen URL, it reuses the already reopened browser tab instead of creating a duplicate.
2. If it creates a new Chrome group for the recovered tab, it applies the Chrome Flow role title.

Expected event evidence:

```text
workspace_tab_readded
recovered_tab_group_restored
recovery_group_restored
groupTitle: <Role · WorkspaceToken>
groupedTabs == totalTabs
ungroupedTabs: 0
pendingActionTraces: []
```

## Validation Checklist

1. Pull latest and reload the extension.
2. Start with 3–5 workspace tabs.
3. Assign roles.
4. Create Chrome Tab Groups.
5. Use `Remove from Workspace` on a grouped tab.
6. Confirm the tab is removed from the workspace.
7. Confirm the browser tab closes.
8. Confirm the Chrome group no longer contains that tab.
9. Use Recovery Journal → `Re-add to Workspace`.
10. Confirm the tab is reopened if needed.
11. Confirm it is restored to the correct role group.
12. Confirm the group title is visible and named correctly.
13. Test `Close Browser Tab` separately.
14. Test `Reopen URL` separately and confirm it only opens the URL.
15. Then click `Re-add to Workspace` and confirm it reuses that reopened tab, restores it to the workspace, and restores titled grouping.
16. Copy workspace and diagnostic packets.
