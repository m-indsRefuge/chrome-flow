# V0.2O Patch — Base Intake Restoration Checkpoint

## Purpose

This checkpoint restores the validated base tab-intake runtime path before continuing advanced-control refinement.

A validation packet showed that after `tabs_scanned`, the next add event was `active_tab_added`, not `selected_tabs_added`. That means the selected scanned-tab intake path did not execute in that pass. The resulting workspace had only one tab because the active-tab path was triggered.

## Immediate Safety Action

The advanced tab controls module is temporarily unloaded from `sidepanel.html`.

The runtime path is now back to:

```text
diagnostics.js
sidepanel.js
workspace-session-control.js
```

The advanced files remain in the repo, but they are not loaded until the base intake path validates again.

## Intake Label Clarification

The intake buttons were relabelled and reordered:

```text
Scan Current Window Tabs
Select All Scanned Tabs
Deselect All Scanned Tabs
Add Selected Scanned Tabs to Workspace
Clear Scanned Tabs
Add Current Active Tab Only
```

This makes the selected scanned-tab path visually distinct from the active-tab-only path.

## Expected Base Behavior

After scanning tabs and selecting multiple scanned tabs:

- clicking Add Selected Scanned Tabs to Workspace should add all selected scanned tab instances
- the scanned tab capture should clear after the add
- System Journal should record `selected_tabs_added`
- Diagnostic trace should record `addSelectedTabsToWorkspace`
- workspace packet should contain all added selected tabs

## Advanced Layer Plan

Do not rewrite the base controller yet.

Recommended sequence:

```text
1. Revalidate base selected-tab intake.
2. Confirm workspace packet creation remains structurally correct.
3. Reattach the refined advanced module above the base layer.
4. Validate advanced controls again.
5. Only after both layers validate, consider native integration.
```
