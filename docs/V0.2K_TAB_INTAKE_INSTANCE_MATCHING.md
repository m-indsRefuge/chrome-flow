# V0.2K — Tab Intake Instance Matching and Duplicate URL Awareness

## Purpose

This slice fixes a tab intake matching issue where Chrome Flow treated different browser tabs with the same URL as if they were the same workspace tab.

The intended distinction is:

- Browser tab identity: the exact open Chrome tab instance.
- Page URL identity: the page/document loaded inside a tab.

Two tabs may have the same URL but still be different browser tab instances. Chrome Flow should allow the user to add both when that is useful to the workspace.

## Behavior

Done when:

- Tab intake distinguishes browser tab identity from page URL identity.
- Exact same browser tab instances remain disabled in Workspace Intake.
- Same-URL tabs with different tab IDs remain selectable.
- Same-URL duplicate candidates show a Same URL already in workspace badge.
- Add Selected Tabs skips exact existing tab instances only.
- Add Selected Tabs allows intentional same-URL duplicate tab instances.
- Selected-tab system events record instance-aware intake metadata.
- Developer diagnostics record the instance-aware intake mode and result counts.

## Implementation Note

The fix is implemented as a side panel intake module:

```text
src/sidepanel/intake-instance-matching.js
```

The module intercepts Add Selected Tabs, uses exact tab ID matching for skips, and treats same-URL matches as a visible warning rather than a blocking condition.

## Validation Scenario

1. Clear workspace tabs.
2. Open two different Chrome tabs to the same URL.
3. Scan Current Window Tabs.
4. Confirm the first exact existing tab is disabled only when it is already in the workspace.
5. Confirm a same-URL but different tab remains selectable.
6. Select both same-URL tabs.
7. Click Add Selected Tabs to Workspace.
8. Confirm both same-URL tab instances are added.
9. Confirm the status mentions same-URL duplicate tabs were allowed.
10. Copy a diagnostic packet and confirm instance-aware intake metadata appears.
